# AGENTS.md

## 项目定位

- 这是一个 Godot 4.6.x 项目，当前使用 GDScript 作为默认协作语言，项目仍处于起步阶段。
- 当前已确认的事实以 `project.godot`、场景、脚本和资源现状为准；如果后续文档与代码不一致，优先按代码和场景实际状态判断，并主动指出差异。
- 项目名称当前为 `food_cleaner`；在核心玩法、主循环、关卡结构尚未稳定前，不要擅自把某种玩法假设写死到实现或文档里。
- 如果后续明确了玩法方向、主入口场景、核心系统和数据流，应及时补充到本文件，而不是继续长期依赖“默认猜测”。

## 必读上下文

开始复杂任务前，优先按这个顺序阅读：

1. `AGENTS.md`：确认协作规则、编辑边界和验证方式。
2. `devlog.md` 最近记录：确认最近改动、验证方式和已知风险。
3. `README.md` 或其他项目说明：确认项目目标、运行方式和当前阶段。
4. `project.godot`：确认 Godot 版本特征、项目名和基础配置。
5. 相关场景、脚本、资源和配置：确认真实实现状态。
6. 如果后续建立了 `docs/`，优先阅读与当前任务直接相关的设计文档、结构文档和开发计划。

简单改动可以只读 `AGENTS.md`、`devlog.md` 最近记录、`project.godot` 和相关脚本/场景，但不要跳过上下文判断。

如果项目里暂时没有 `devlog.md`、`README.md` 或 `docs/`，要明确说明“文档缺失，当前判断主要基于代码与资源现状”。

## 目录与职责

当前项目还很早期，目录职责应以现状为准。默认按以下原则理解和扩展：

- `project.godot`：项目配置真源，用于确认项目名、渲染特性和基础设置。
- `scenes/`：主场景、关卡场景、UI 场景等。后续如建立子目录，应按功能拆分而不是混放。
- `scripts/`：GDScript 脚本。优先按系统职责分目录，不要把所有脚本堆在根目录。
- `assets/` 或 `art/`：美术资源。如果项目后续采用其他资源目录命名，沿用现有约定。
- `audio/`：音频资源；没有该目录时不要为了“整齐”强行创建。
- `addons/`：第三方插件或工具插件。除非任务明确涉及插件，不要改。
- `.godot/`：Godot 编辑器生成目录。不要主动编辑或整理。
- `.vscode/`：编辑器工作区配置。除非任务明确涉及开发工具配置，否则不要顺手修改。

如果项目后续形成稳定结构，应把这里改成基于真实目录的说明，而不是继续保留模板描述。

## 实现原则

- 先理解现有结构、命名、信号连接方式、场景组织和脚本职责，再动手修改。
- 优先沿用项目已有模式，不要为了小改动引入新架构、新依赖或大范围重构。
- 在起步阶段尤其要控制“先搭大框架”的冲动；没有明确收益时，优先做小而清晰、便于演进的实现。
- 文件名使用 `snake_case`。
- GDScript 类名使用 `PascalCase + class_name`，但仅在项目已有该习惯或确实需要全局注册时使用。
- 注释优先解释规则语义、状态变化、副作用和边界条件，不写低信息量注释。
- UI、输入、碰撞、存档、关卡切换等改动，优先复用 Godot 现有节点能力和项目既有写法，不要过早自造复杂系统。
- 文本和 UI 改动要注意中英文长度差异、按钮可点击区域、分辨率适配和窗口缩放下的可读性。

## 编辑规则

- 改文件前先说明准备改什么、为什么改。
- 默认按任务风险控制改动范围；如果改动会影响主循环、场景切换、输入、资源加载、存档或核心节点结构，要先说明影响面和验证方式。
- 不修改与当前任务无关的格式、配置、导入资源或编辑器生成文件。
- 不覆盖、回滚或删除用户已有未提交改动。
- 如果发现更深层问题，先说明判断、风险和建议，再决定是否扩大修改范围。
- 机制、流程、UI 文案、资源组织约定或验证方式发生变化时，同步更新相关文档和 `devlog.md`。
- `devlog.md` 记录应包含：
  - 做了什么
  - 影响文件
  - 如何验证
- 如果只是在起草方案，或用户明确要求“不要改文件”，不得创建、覆盖或移动文件。

## 验证规则

完成代码、场景或配置相关修改后，优先做与改动匹配的验证：

- 静态错误检查：优先检查受影响脚本和场景，再按风险决定是否做全项目检查。
- 默认不依赖 Godot MCP；优先使用 headless 启动、项目自带脚本、静态检查和必要的手动回归进行验证。
- 当前项目的基础 headless 启动命令可用：
  - `D:\Godot\Godot_v4.6.2-stable_win64.exe --headless --path D:\mygodot\food-cleaner --quit`
- 如果项目后续引入了测试场景、自动化脚本或命令行测试入口，优先补充到本节，并在任务完成后按改动范围执行。
- 涉及以下内容时，除基础启动外尽量补做针对性回归：
  - 主场景/启动流程
  - 输入响应
  - 场景切换
  - UI 交互
  - 资源加载与导出相关配置
  - 存档、数据持久化或配置读写

如果环境无法运行验证，要明确说明原因，并给出可手动验证步骤。
